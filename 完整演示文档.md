## 四、正式演示流程

## 4.1 开场介绍

### 操作

先停留在管理后台登录页，不进行点击。

### 讲解词

> 各位老师好，我的作品是《开发者留存雷达》，对应命题 B“促进开发者活跃”。高校训练营通常会分别使用问卷、群接龙、在线表格和代码平台记录报名、账号开通、任务完成和最终作品。这些数据彼此分散，运营人员往往只能看到最后提交了多少人，却不知道参与者具体在哪个阶段流失。开发者留存雷达通过统一的报名和签到入口，将每一位参与者在不同阶段的完成记录连接起来，并实时生成留存漏斗，帮助运营人员及时发现问题并采取干预措施。

### 必须传达的信息

- 这是一个真实可使用的 Web 应用，不是静态原型。
- 系统服务于训练营运营场景。
- 核心价值是统一数据、连续追踪和发现流失节点。

## 4.2 登录管理后台

### 操作

1. 展示登录页上的评委演示账号。
2. 输入密码。
3. 点击“登录后台”。
4. 进入“数据概览”。

### 讲解词

> 参与者不需要注册账号，管理员使用单独的管理账号登录。管理员会话通过 HttpOnly Cookie 保存，最长有效时间为 8 小时。登录页面直接提供评委演示账号，方便作品验收。

### 可顺带展示

- 点击右上角月亮按钮切换夜间模式。
- 再点击一次恢复日间模式。
- 说明主题选择会保存到浏览器，下次打开仍然保留。

## 4.3 展示数据概览

### 操作

1. 确认右上角选中“2026 华为云开发者实战营”。
2. 指向顶部四个数据卡片。
3. 指向实时转化漏斗。
4. 指向右侧阶段转化列表。
5. 指向顶部“重点关注”提示。

### 讲解词

> 当前活动共有 100 人报名，其中 80 人完成账号开通，50 人完成首个任务，35 人完成最终提交。系统会自动计算每一步相对于上一步的转化率。账号开通转化率为 80%，首个任务转化率为 62.5%，最终提交转化率为 70%，整体完成率为 35%。系统还会自动指出转化率最低的阶段，运营人员可以针对这一阶段增加提醒、辅导或激励。

### 数据口径

- 阶段人数是该阶段去重后的参与者人数。
- 阶段转化率等于当前阶段人数除以上一阶段人数。
- 整体完成率等于最终阶段人数除以报名阶段人数。
- 同一个人重复签到不会重复计数。

### 展示日期筛选

1. 在“数据时间”中选择开始日期和结束日期。
2. 等待数据刷新。
3. 说明日期筛选按签到记录时间统计。
4. 点击“清除”恢复全部数据。

### 展示图表导出

1. 点击“导出图表”。
2. 说明系统会下载漏斗 PNG。
3. 该图片可以直接用于运营周报、总结材料和答辩 PPT。

## 4.4 展示统一签到入口

### 操作

1. 点击左侧“签到入口”。
2. 展示四个阶段的二维码卡片。
3. 指出每个阶段都有独立二维码和链接。
4. 点击一个阶段的“复制链接”。
5. 展示“已复制”反馈。
6. 点击“打开入口”展示电脑端签到页面，然后返回后台。

### 讲解词

> 每个任务阶段都有独立入口。管理员可以投屏二维码，也可以将链接发送到微信群或 QQ 群。参与者无需注册系统账号，直接扫码即可操作。复制功能同时兼容 HTTPS 和当前 HTTP 演示环境，即使浏览器限制自动剪贴板，页面也会显示可手动选择的完整链接。

### 强调

- “活动报名”用于创建参与者档案。
- “账号开通”和“完成首个任务”属于普通签到。
- “最终提交”会额外要求填写作品链接。
- 不同阶段不能共用同一二维码。

## 4.6 展示同一参与者的身份匹配

### 操作

1. 返回“签到入口”。
2. 用手机扫描“账号开通”二维码。
3. 填写与报名时相同的学号和手机号。

### 讲解词

> 系统在当前活动范围内通过学号或手机号匹配参与者。任意一项与报名档案一致，就会认为这是同一个人，并将新阶段记录关联到原参与者。不同活动的数据相互隔离，所以相同学号在另一个活动中不会串联。

### 预期数据变化

```text
活动报名：101
账号开通：80 → 81
完成首个任务：50
最终提交：35
```

### 身份规则

- 学号和手机号在同一活动中分别唯一。
- 身份字段会去除空格并统一大小写。
- 姓名不是主要匹配键。
- 后续阶段必须先有报名档案。

## 4.7 展示防重复签到

### 操作

1. 手机再次打开刚才的“账号开通”入口。
2. 使用同一身份再次提交。
3. 展示“你已完成过本阶段”。
4. 回到后台刷新。
5. 确认账号开通人数仍然是 81，没有变成 82。

### 讲解词

> 同一参与者在同一阶段只能产生一条签到记录。重复提交仍然返回友好的成功状态，避免参与者误以为失败，但不会增加统计人数。这一规则由数据库唯一约束保证，不只是前端判断。

## 五、评委常见问题与回答

### 5.1 系统如何判断不同步骤是同一个人？

回答：

> 参与者第一次报名时会建立档案。后续阶段在当前活动中通过学号或手机号匹配，任意一项相同就认为是同一个人。不同活动的数据相互隔离，同一活动中的学号和手机号分别唯一。

### 5.2 为什么参与者不需要登录？

回答：

> 训练营签到强调低操作门槛。如果要求每位参与者注册账号，会增加流失。系统使用活动专属二维码和身份字段匹配，在便捷性与数据连续性之间取得平衡。管理员后台仍然使用独立认证保护。

### 5.3 能否防止代签？

回答：

> 当前版本属于运营统计工具，不是强实名考试系统。它通过学号或手机号建立身份连续性，但知道他人信息的人理论上可能代签。正式业务可以继续接入短信验证码、学校统一身份认证或华为云身份服务增强认证强度。

### 5.4 为什么重复提交仍然显示成功？

回答：

> 这是幂等设计。参与者可能因为网络延迟重复点击，如果直接报错会造成困惑。系统返回已完成提示，但数据库唯一约束保证人数不会增加。

### 5.5 数据是实时更新的吗？

回答：

> 每次签到写入数据库后即可查询。后台支持手动刷新，并且每 30 秒自动拉取最新数据，因此运营人员不需要重新打开页面。

### 5.6 演示数据为什么没有大连理工大学？

回答：

> 演示数据来自其余 38 所 985 高校，特意排除了大连理工大学。这样现场扫码产生的大连理工大学真实测试数据可以与预设演示数据清楚区分。

### 5.7 演示数据每次都不一样吗？

回答：

> 姓名、学校、学号、手机号、时间和进度最初使用随机算法生成，但保存了固定随机种子，所以每次重置都会稳定复现同一批数据，既具有真实分布，也方便验收和复盘。

### 5.8 如何计算转化率？

回答：

> 阶段转化率等于当前阶段去重人数除以上一阶段去重人数，整体完成率等于最后阶段人数除以报名阶段人数，百分比保留两位小数。

### 5.9 活动关闭后数据会丢失吗？

回答：

> 不会。关闭只禁止公开入口继续写入，后台历史漏斗、参与者、签到时间和作品链接仍可查看和导出。

### 5.10 删除阶段后历史记录怎么办？

回答：

> 阶段采用软删除，前台和正常后台列表不再显示，但历史审计数据不会直接被物理删除。

### 5.11 系统如何保护数据？

回答：

> 管理后台使用 JWT HttpOnly Cookie；接口进行输入校验和限流；数据库不开放公网端口；敏感配置通过环境变量注入；生产环境可进一步配置 HTTPS、强密码和定期备份。

### 5.12 为什么选择 PostgreSQL？

回答：

> 活动、阶段、参与者和签到之间具有明确关系，并且需要唯一约束和事务保证重复保护与排序一致性。PostgreSQL 很适合这类结构化运营数据。

### 5.13 系统如何扩展？

回答：

> 可以继续增加短信提醒、微信通知、学校统一认证、多个管理员角色、参与者批量导入、运营干预记录和更细粒度的数据分析。当前 REST API 和动态阶段模型已经为这些扩展提供基础。

## 六、现场异常处理

### 6.1 后台自动登录，没有显示登录页

原因：浏览器保留了最长 8 小时的有效会话。

处理：点击左下角“退出登录”，再重新打开 `/admin`。

### 6.2 登录后仍显示旧界面

处理：

1. 按 `Ctrl + F5`。
2. 关闭标签页重新打开。
3. 必要时清除该网站缓存。

### 6.3 手机扫二维码打不开

检查：

- 手机是否联网。
- 二维码是否来自公网地址打开的后台。
- 地址是否为 `123.60.220.148`，而不是 `localhost` 或内网 IP。
- 华为云安全组是否放行 80 端口。

替代方式：点击“复制链接”，把链接发送到手机打开。

### 6.4 复制链接按钮没有反应

当前系统已经提供三重处理：

1. 安全剪贴板复制。
2. HTTP 环境兼容复制。
3. 浏览器仍然阻止时弹出完整链接供手动复制。

每张卡片也永久显示完整链接，并提供“打开入口”。

### 6.5 报名提示格式错误

检查：

- 姓名是否至少 2 个字符。
- 学号是否至少 3 个字符。
- 手机号是否只包含数字。
- 作品链接是否是完整 URL。

### 6.6 提示“活动当前未开放”

进入“活动设置”，确认活动状态为“进行中”。

### 6.7 提示“未找到报名信息”

先扫描当前活动的报名二维码，再使用相同学号或手机号完成后续阶段。

### 6.8 后台人数没有变化

1. 点击“刷新数据”。
2. 清除日期筛选。
3. 检查当前选择的活动。
4. 确认手机操作已经显示成功。

### 6.9 页面无法访问

服务器执行：

```bash
cd /opt/developer-retention-radar
docker compose ps
docker compose logs --tail=100 web
docker compose logs --tail=100 api
curl http://localhost/api/health
```

### 6.10 现场网络完全不可用

不要假装在线功能正常。可以依次展示：

1. 提前准备的后台和手机端截图。
2. 导出的漏斗 PNG。
3. CSV 示例。
4. Git 仓库中的代码、测试和 Docker 配置。
5. 说明公网服务此前已完成验证，并展示部署证明材料。

## 七、演示过程中的禁止操作

- 不要在需要保留数据时点击“重置演示数据”。
- 不要删除当前演示活动。
- 不要随意修改阶段类型后忘记恢复。
- 不要把最终提交阶段移动到报名阶段之前。
- 不要展示服务器 SSH 密码、数据库密码、访问令牌或 `.env`。
- 不要在公开画面中长时间展示真实参与者完整手机号。
- 不要使用已有演示学号测试新增报名。
- 不要使用 `git reset --hard` 处理服务器代码。
- 不要对公网开放 PostgreSQL 5432 端口。

## 八、演示结束后的恢复操作

完成答辩后：

1. 确认活动状态恢复为“进行中”或按实际需要设为“已结束”。
2. 退出管理后台，特别是在公共电脑上。
3. 如需保留现场记录，导出 CSV 和漏斗图。
4. 如不需要现场测试记录，可在确认无真实数据后重置演示数据。
5. 检查公网地址仍可访问。
6. 检查代码已提交并推送。
7. 检查最终 ZIP 包包含最新代码和文档。
