## 一、系统简介

开发者留存雷达是一套面向高校训练营、开发者活动和课程实训的报名、签到、作品提交与留存分析系统。

传统训练营常使用多个问卷、群接龙和表格分别记录报名、账号开通、任务完成和最终作品，运营人员难以判断参与者在哪一步流失。本系统为每一个任务阶段生成独立二维码和链接，让参与者通过同一套身份信息完成连续签到，再在后台形成阶段漏斗。

系统主要解决以下问题：

- 将报名、账号开通、任务签到和最终提交统一到同一套系统中。
- 使用学号或手机号匹配参与者，形成连续的完成轨迹。
- 自动避免重复签到导致的统计膨胀。
- 自动计算阶段转化率和整体完成率。
- 快速定位转化率最低的阶段，辅助运营干预。
- 查看参与者明细、作品链接并导出 CSV。
- 通过手机二维码实现免登录报名和签到。

### 1.1 主要技术

- 前端：React、TypeScript、Vite。
- 图表：ECharts。
- 二维码：QRCode React。
- 后端：Node.js、Express、TypeScript。
- 数据库：PostgreSQL。
- 认证：JWT HttpOnly Cookie。
- 部署：Docker Compose、Nginx。
- 云端环境：华为云 Flexus 应用服务器 L 实例。

### 1.2 浏览器建议

- 电脑端推荐最新版 Edge、Chrome 或 Firefox。
- 手机端可使用系统浏览器、微信内置浏览器或 QQ 内置浏览器。
- 如果页面仍显示旧内容，请刷新页面；手机内置浏览器必要时关闭页面后重新扫码。

## 二、核心概念和数据口径

### 2.1 训练营或活动

一个活动代表一次独立的训练营、课程、比赛或运营项目。不同活动的参与者和签到记录相互隔离。

活动包含三个状态：

| 状态 | 公开报名/签到 | 后台查看 | 典型用途 |
| --- | --- | --- | --- |
| 草稿 | 不允许 | 允许 | 配置阶段、检查二维码 |
| 进行中 | 允许 | 允许 | 正式报名、签到和作品提交 |
| 已结束 | 不允许 | 允许 | 活动归档、保留历史数据 |

只有“进行中”的活动允许参与者提交。新活动创建后默认为“草稿”，必须由管理员手动切换为“进行中”。

### 2.2 任务阶段

每个活动由若干有顺序的任务阶段组成。系统支持三种阶段类型：

| 阶段类型 | 用途 | 行为 |
| --- | --- | --- |
| 报名 | 建立参与者档案 | 可以创建或更新本活动中的参与者信息 |
| 普通签到 | 记录账号开通、任务完成等进度 | 必须匹配本活动中已有的报名信息 |
| 作品提交 | 收集最终作品、代码或演示链接 | 必须先报名，并且必须填写有效网址 |

创建新活动时，系统自动生成四个默认阶段：

1. 活动报名。
2. 账号开通。
3. 完成首个任务。
4. 最终提交。

管理员可以编辑阶段名称和说明、改变阶段类型、添加新阶段、软删除阶段，并通过上下箭头调整顺序。

### 2.3 参与者

参与者不需要注册系统账号。参与者通过姓名、学号、手机号和可选的学校信息完成报名。

在同一个活动中：

- 学号必须唯一。
- 手机号必须唯一。
- 后续签到可以通过学号或手机号匹配已有参与者。
- 系统会去除身份字段首尾空格、内部空白并统一大小写后再匹配。

### 2.4 签到记录

每次成功完成某个阶段会产生一条签到记录，记录包含：

- 阶段。
- 参与者。
- 签到时间。
- 来源，当前公开页面来源为 `web`。
- 浏览器设备信息。
- 作品链接（仅作品提交阶段使用）。

同一个参与者在同一个阶段只能产生一条有效记录。重复提交仍返回成功页面，但不会增加人数。

### 2.5 统计口径

- 阶段人数：在该阶段具有签到记录的去重参与者人数。
- 阶段转化率：当前阶段人数 ÷ 前一阶段人数 × 100%。
- 第一阶段转化率固定显示为 100%。
- 整体完成率：最后一个阶段人数 ÷ 第一个阶段人数 × 100%。
- 分母为 0 时，转化率显示为 0%。
- 百分比最多保留两位小数。
- 日期筛选按签到记录的创建时间统计，不改变参与者原始档案。

示例：阶段人数为 `100 → 80 → 50 → 35` 时：

- 账号开通转化率为 80%。
- 首次任务转化率为 62.5%。
- 最终提交转化率为 70%。
- 整体完成率为 35%。

## 三、角色与访问方式

### 3.1 管理员

管理员负责创建活动、设置阶段、开放或关闭活动、查看统计数据以及导出参与者名单。

访问地址：

```text
http://123.60.220.148/admin
```

当前比赛演示环境的账号显示在登录页面：

```text
用户名：admin
密码：Radar-c465837015ea6dcf2c38
```

生产使用时，应通过服务器 `.env` 修改管理员密码，不建议长期使用演示密码。

登录成功后会在浏览器中保存最长 8 小时的 HttpOnly 会话 Cookie。因此，短时间内再次打开后台可能会自动登录，这是正常现象。点击左下角“退出登录”可以立即清除当前会话。

### 3.2 参与者

参与者无需登录。管理员从“签到入口”页面展示二维码或复制链接，参与者扫码或打开链接即可操作。

公开链接格式：

```text
http://服务器地址/checkin/阶段ID
```

参与者不应从网站首页自行寻找活动，因为不同活动和不同阶段使用不同链接。应使用管理员提供的对应二维码。

### 3.3 评委或演示人员

评委既可以使用演示账号查看后台，也可以使用手机扫描二维码体验报名、签到和作品提交。

## 四、管理员快速上手

如果系统已经部署完成，最短使用流程如下：

1. 打开管理后台 `/admin`。
2. 输入管理员账号和密码登录。
3. 点击右上角“新建活动”。
4. 填写活动名称和活动说明。
5. 点击“创建并生成默认阶段”。
6. 在右上角活动下拉框中确认已选中刚创建的活动。
7. 进入“活动设置”，检查四个默认阶段。
8. 将活动状态从“草稿”切换为“进行中”。
9. 进入“签到入口”，找到“活动报名”二维码。
10. 用手机扫码，填写姓名、学号、手机号和学校。
11. 点击“确认报名”。
12. 返回后台，点击“刷新数据”或等待最多 30 秒自动刷新。
13. 在“数据概览”中查看报名人数，在“参与者”中查看个人记录。

## 五、创建并开放一个新活动

### 5.1 新建活动

1. 登录管理后台。
2. 点击页面右上角“新建活动”。
3. 在“活动名称”中填写至少 2 个字符，例如“2026 华为云数据库实战营”。
4. 在“活动说明”中填写活动目标、时间或面向人群。
5. 点击“创建并生成默认阶段”。

系统会创建活动，并依次生成“活动报名”“账号开通”“完成首个任务”“最终提交”四个阶段。

### 5.2 选择活动

后台右上角的下拉框用于切换当前管理的活动。所有页面中的漏斗、二维码、参与者和设置都属于当前选择的活动。

如果新建后没有看到活动内容：

1. 打开右上角活动下拉框。
2. 手动选中刚创建的活动。
3. 点击“刷新数据”。

### 5.3 检查和修改阶段

进入“活动设置”，在“任务阶段”区域可以：

- 点击“编辑”修改阶段名称、说明和类型。
- 点击“添加阶段”增加新的任务节点。
- 点击 `↑` 或 `↓` 调整阶段顺序。
- 点击“删除”将阶段软删除。

阶段排序会直接影响漏斗显示顺序和整体完成率的“最终阶段”。因此，报名阶段应放在第一位，作品提交阶段通常放在最后一位。

阶段类型设置错误会导致行为变化：

- 设为“报名”时，该入口可以创建参与者。
- 设为“普通签到”时，该入口只能匹配已有参与者。
- 设为“作品提交”时，页面会要求填写有效链接。

### 5.4 开放活动

新活动默认处于“草稿”状态。进入“活动设置”，点击“进行中”。

只有完成这一步后，手机端才能报名或签到。草稿或已结束状态下，手机端会显示“活动当前未开放，暂时不能签到”。

### 5.5 发布报名入口

1. 进入“签到入口”。
2. 找到名称为“活动报名”、类型为报名的第一阶段。
3. 直接投屏展示二维码，或点击“复制链接”。
4. 将链接发送到微信群、QQ群或课程通知中。
5. 发布前使用手机流量测试一次，确认公网可访问。

不要把“账号开通”“完成首个任务”或“最终提交”的二维码当作报名二维码。未报名人员直接扫描后续阶段会收到“未找到报名信息，请先完成报名”。

## 六、参与者报名和签到

### 6.1 首次报名

参与者操作步骤：

1. 使用手机扫描管理员提供的“活动报名”二维码。
2. 确认页面顶部显示的是正确活动，标题为“活动报名”或管理员设置的报名名称。
3. 填写真实姓名。
4. 填写学号。
5. 填写手机号。
6. 选填学校。
7. 点击“确认报名”。
8. 页面显示“签到成功”即报名完成。

字段规则：

| 字段 | 是否必填 | 规则 |
| --- | --- | --- |
| 姓名 | 是 | 2～40 个字符 |
| 学号 | 是 | 3～40 个字符 |
| 手机号 | 是 | 6～15 位数字 |
| 学校 | 否 | 最多 80 个字符 |

当前系统主要面向中国大陆手机号，建议填写标准 11 位手机号。

### 6.2 后续普通签到

参与者完成账号开通或任务后：

1. 扫描该任务对应的二维码。
2. 填写姓名、已报名的学号和手机号。
3. 点击“完成签到”。
4. 页面显示“签到成功”。

后续阶段不会创建新参与者。学号或手机号至少有一项必须能匹配当前活动中的报名档案。

### 6.3 最终作品提交

1. 扫描“最终提交”二维码。
2. 填写已报名的身份信息。
3. 在“作品链接”中填写完整网址。
4. 点击“提交作品”。

有效链接示例：

```text
https://atomgit.com/example/project
https://github.com/example/project
https://example.com/demo
```

链接必须包含 `http://` 或 `https://`。仅填写仓库名称、网盘提取码或普通文字会被判定为格式错误。

### 6.4 重复报名或签到

同一参与者重复提交同一阶段时，页面会显示“你已完成过本阶段”，系统不会重复计数。

报名阶段再次使用相同学号提交时，会更新该参与者的姓名、手机号和学校信息，然后保持该阶段只有一条签到记录。

### 6.5 报名后在哪里查看

管理员切回后台后：

- 在“数据概览”查看报名人数变化。
- 在“参与者”查看姓名、学号、手机号、学校和已完成阶段数量。
- 点击右上角“刷新数据”立即拉取最新数据。
- 不手动刷新时，系统每 30 秒自动更新一次。

## 七、管理后台完整功能

### 7.1 顶部区域

顶部区域包括：

- 当前活动名称。
- 数据最近刷新时间。
- 活动选择下拉框。
- “刷新数据”按钮。
- “新建活动”按钮。

管理多个活动时，执行任何操作前都应先确认下拉框中选择的是正确活动。

### 7.2 数据概览

“数据概览”包含：

- 报名人数。
- 最终完成人数。
- 整体完成率。
- 任务阶段数量。
- 实时转化漏斗。
- 各阶段人数和相对上一阶段的转化率。
- 转化率最低阶段的重点关注提示。

#### 日期筛选

可以选择开始日期和结束日期，系统只统计该时间范围内产生的签到记录。

- 只填开始日期：统计该日期以后数据。
- 只填结束日期：统计该日期及以前数据。
- 同时填写：统计闭区间内数据。
- 点击“清除”：恢复全部时间范围。

日期筛选只作用于漏斗统计，不会删除数据。

#### 导出图表

漏斗有数据时，点击“导出图表”会下载 PNG 图片，默认文件名为：

```text
retention-funnel.png
```

可将图片用于周报、答辩 PPT 或运营复盘。

### 7.3 签到入口

系统为每个未删除阶段展示一张二维码卡片，卡片包括：

- 阶段顺序。
- 阶段名称。
- 阶段说明。
- 二维码。
- “复制链接”按钮。

二维码内容根据当前浏览器访问的域名自动生成。因此，必须通过公网地址打开后台后再展示二维码，避免二维码中出现 `localhost` 或内网地址。

如果浏览器拒绝剪贴板权限，点击“复制链接”可能失败。此时可以手动复制浏览器中打开后的签到地址，或在浏览器设置中允许该网站使用剪贴板。

### 7.4 参与者

“参与者”页面显示：

- 姓名和学校。
- 学号。
- 手机号。
- 完成阶段数量。
- 最终作品链接。

#### 搜索

搜索框支持匹配姓名、学号和手机号。输入内容后在当前已加载名单中即时筛选。

#### 阶段筛选

阶段下拉框可以只显示完成了某一阶段的参与者。例如选择“最终提交”，即可查看最终完成人员。

#### 查看作品

参与者提交作品后，“作品链接”列会显示“查看作品”。点击后在新标签页打开。

#### 导出 CSV

点击“导出 CSV”可下载当前活动的完整参与者名单。文件包含：

- 姓名。
- 学号。
- 手机号。
- 学校。
- 报名时间。
- 作品链接。

CSV 使用 UTF-8 BOM，通常可直接用 Microsoft Excel 正确显示中文。

当前导出为活动完整名单，不受页面搜索词和阶段筛选条件影响。

### 7.5 活动设置

#### 活动状态

- 草稿：准备阶段，禁止公开提交。
- 进行中：正式开放。
- 已结束：关闭公开提交并保留历史数据。

#### 演示数据重置

点击“重置演示数据”并确认后，系统会：

1. 清空全部活动。
2. 清空全部阶段。
3. 清空全部参与者。
4. 清空全部签到和作品数据。
5. 创建一个新的演示活动。
6. 生成来自其余 38 所 985 高校（特意不含大连理工大学）的固定随机参与者数据，并呈现 `100 → 80 → 50 → 35` 的示范漏斗。姓名、学校、学号、手机号、签到时间和完成进度在首次设计时随机生成，但使用固定随机种子，之后每次重置都会稳定复现同一批演示数据。这样可以将演示数据与大连理工大学真实报名人员清楚区分。

**这是全局破坏性操作，不是只重置当前活动。真实运营环境中严禁随意使用。**

#### 阶段编辑和排序

编辑阶段后，已有签到记录仍与该阶段关联。删除阶段采用软删除，后台不再显示该阶段，但数据库中的历史签到审计记录仍保留。

调整阶段顺序后，漏斗顺序和转化率会随之变化。

### 7.6 退出登录

点击左下角“退出登录”。公共电脑演示结束后应主动退出，避免其他人继续访问参与者信息。

## 八、标准运营流程

### 8.1 活动开始前

1. 新建活动。
2. 检查默认阶段。
3. 按实际安排新增或修改阶段。
4. 确保报名阶段位于第一位。
5. 确保最终提交阶段位于最后一位。
6. 使用公网地址打开后台。
7. 将状态切换为“进行中”。
8. 手机实测报名二维码。
9. 准备参与者隐私告知。
10. 发布报名链接。

### 8.2 活动进行中

1. 每个任务节点只发布对应阶段二维码。
2. 观察“数据概览”的人数和转化率。
3. 对转化率最低阶段进行提醒或辅导。
4. 使用“参与者”筛选已完成和未完成情况。
5. 必要时导出 CSV 进行联系和复盘。

### 8.3 活动结束后

1. 确认最终作品均已提交。
2. 导出参与者 CSV。
3. 导出漏斗 PNG。
4. 将活动状态切换为“已结束”。
5. 备份 PostgreSQL 数据库。
6. 保留后台用于历史查询。

## 九、演示数据和比赛答辩

### 9.1 答辩前准备

1. 确认服务器和三个应用容器正常运行。
2. 电脑打开管理后台。
3. 手机关闭 Wi-Fi，使用移动网络扫码测试。
4. 准备一个未使用的学号和手机号。
5. 如使用演示数据，提前执行重置并确认漏斗为 `100 → 80 → 50 → 35`。
6. 检查 AtomGit 仓库和云端 URL 可以打开。
7. 不要在正式数据环境执行演示重置。

### 9.2 推荐 5 分钟演示顺序

#### 第 1 步：介绍问题（约 30 秒）

说明报名、签到和作品提交分散在多个工具中，运营人员难以及时识别流失节点。本系统将全过程统一为连续留存漏斗。

#### 第 2 步：展示漏斗（约 60 秒）

打开“数据概览”，讲解阶段人数、转化率、整体完成率和最低转化节点提示，并演示日期筛选和图表导出。

#### 第 3 步：展示报名入口（约 60 秒）

打开“签到入口”，投屏二维码，使用手机扫描“活动报名”，填写新身份并提交。返回后台点击“刷新数据”，展示报名人数变化。

#### 第 4 步：展示连续签到（约 45 秒）

扫描“账号开通”或“完成首个任务”二维码，使用相同学号完成签到。再次提交同一阶段，展示防重复机制。

#### 第 5 步：展示作品提交和名单（约 60 秒）

扫描“最终提交”二维码，提交作品链接。进入“参与者”，展示作品留档、搜索、阶段筛选和 CSV 导出。

#### 第 6 步：展示管理能力（约 45 秒）

进入“活动设置”，展示阶段编辑、排序和活动关闭。说明已结束活动禁止公开签到，但后台历史数据仍保留。

### 9.3 推荐结束语

> 开发者留存雷达把分散、滞后的人工统计变成统一入口的连续数据采集，使训练营运营人员能够快速定位流失阶段并及时干预，从而促进开发者持续参与华为云生态中的学习、探索与贡献。

### 10.1 环境要求

- Node.js 22 或更高版本。

- PostgreSQL 15 或更高版本。

```bash
git clone https://atomgit.com/GeYugong/developer-retention-radar.git
```

如果仓库为私有仓库，需要先配置 AtomGit/GitCode 访问权限或使用有权限的凭据。

```bash
npm install
```

项目使用 npm workspaces，根目录安装会同时安装 API 和 Web 依赖。

### 10.4 配置环境变量

复制示例文件：

```bash
cp .env.example .env
```

Windows PowerShell：

```powershell
Copy-Item .env.example .env
```

主要变量：

| 变量 | 用途 | 示例 |
| --- | --- | --- |
| `POSTGRES_DB` | 数据库名 | `radar` |
| `POSTGRES_USER` | 数据库用户 | `radar` |
| `POSTGRES_PASSWORD` | 数据库密码 | 强随机密码 |
| `JWT_SECRET` | 管理员会话签名密钥 | 至少 32 位随机字符串 |
| `ADMIN_USERNAME` | 管理员用户名 | `admin` |
| `ADMIN_PASSWORD` | 管理员密码 | 至少 8 位强密码 |
| `PUBLIC_URL` | 公网访问根地址 | `http://localhost` |

API 还支持直接设置：

| 变量 | 默认值 | 说明 |

| `NODE_ENV` | `development` | `development`、`test` 或 `production` |
| `PORT` | `3000` | API 监听端口 |
| `DATABASE_URL` | 本机 PostgreSQL 地址 | 完整连接字符串 |
| `ADMIN_PASSWORD_HASH` | 未设置 | 可选的 bcrypt 密码哈希，设置后优先使用 |

不要将 `.env`、真实密码或访问令牌提交到 Git。项目 `.gitignore` 已忽略 `.env`。

### 10.5 配置前端开发 API 地址

开发模式下可创建 `apps/web/.env.local`：


如果不设置，前端默认请求同源 `/api`，适合 Nginx 生产部署。

默认地址：

- Web：`http://localhost:5173`
- API：`http://localhost:3000`
- 健康检查：`http://localhost:3000/api/health`

API 启动时会自动执行幂等数据库迁移并创建缺失表和索引。

### 10.7 运行演示数据脚本

```bash
npm run build -w @radar/api
npm run seed -w @radar/api
```

该脚本同样会清空全部业务数据，请仅在开发或演示数据库中使用。

### 10.8 测试和构建

```bash
npm run lint
npm run build
```

- `npm test`：运行 API 单元/集成测试和 Web 组件测试。

## 十一、华为云生产部署

### 11.1 推荐服务器配置

- Ubuntu 22.04/24.04 或兼容 Linux。
- 2 核 CPU。
- 4 GiB 内存。
- 40 GiB 以上系统盘。
- 已安装 Git、Docker Engine 和 Docker Compose 插件。

### 11.2 安全组端口

| 端口 | 是否放行公网 | 用途 |
| --- | --- | --- |
| 22/TCP | 建议仅允许管理员 IP | SSH 管理 |
| 80/TCP | 是 | HTTP 网站 |
| 443/TCP | 配置 HTTPS 后放行 | HTTPS 网站 |
| 5432/TCP | 否 | PostgreSQL 只在 Docker 内部访问 |

### 11.3 克隆并配置

```bash
git clone <仓库地址> /opt/developer-retention-radar
cd /opt/developer-retention-radar
cp .env.example .env
openssl rand -hex 32
```

编辑 `.env`，至少替换：

- `POSTGRES_PASSWORD`。
- `JWT_SECRET`。
- `ADMIN_PASSWORD`。
- `PUBLIC_URL`。

公网 IP 部署示例：

```text
PUBLIC_URL=http://123.60.220.148
```

### 11.4 启动服务

```bash
docker compose config
docker compose build
docker compose up -d
docker compose ps
```

正常情况下包含三个容器：

- `db`：PostgreSQL。
- `api`：Express API。
- `web`：Nginx 和 React 静态页面。

### 11.5 验证部署

```bash
curl http://localhost/api/health
curl http://localhost/health
docker compose ps
```

预期 API 返回：

```json
{"status":"ok"}
```

然后从电脑和手机分别访问公网地址。

### 11.6 HTTPS

生产活动建议使用域名和 HTTPS。可以在华为云 ELB/证书服务或服务器上的 Caddy、Nginx 中终止 TLS。

启用 HTTPS 后：

1. 放行 443 端口。
2. 配置可信证书。
3. 将 `PUBLIC_URL` 改为 `https://你的域名`。
4. 重启 API 和 Web。

HTTPS 环境下管理员 Cookie 会自动启用 `Secure`，降低会话被窃取风险。

## 十三、接口说明

生产环境通常通过同源 `/api` 访问。除公开阶段接口和健康检查外，管理接口均要求管理员登录 Cookie 或 Bearer Token。

### 13.1 健康检查

```http
GET /api/health
```

成功响应：

```json
{"status":"ok"}
```

### 13.2 管理员认证

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/auth/login` | 管理员登录 |
| POST | `/api/auth/logout` | 退出并清除 Cookie |
| GET | `/api/auth/me` | 检查当前登录状态 |

登录请求示例：

```json
{
  "username": "admin",
  "password": "你的管理员密码"
}
```

### 13.3 活动管理

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/campaigns` | 获取全部活动 |
| POST | `/api/campaigns` | 创建活动 |
| PATCH | `/api/campaigns/:id` | 修改名称、说明或状态 |
| DELETE | `/api/campaigns/:id` | 删除活动及关联业务数据 |

活动状态只能是：`draft`、`active`、`closed`。

### 13.4 阶段管理

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/campaigns/:id/stages` | 获取未删除阶段 |

阶段类型只能是：`registration`、`checkin`、`submission`。

### 13.5 公开签到

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/public/stages/:id` | 获取公开阶段信息 |

作品提交阶段必须将 `submissionUrl` 设置为有效 URL。

### 13.6 数据分析和导出

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/campaigns/:id/analytics` | 获取漏斗和完成率 |

```text
?from=2026-07-01&to=2026-07-31
```

参与者接口支持：


## 十四、数据库说明

### 14.1 核心表

| 表 | 作用 |
| --- | --- |
| `campaigns` | 活动信息和状态 |
| `stages` | 阶段、类型、位置和软删除时间 |
| `participants` | 活动中的参与者身份档案 |
| `checkins` | 阶段完成记录、来源、设备和作品链接 |

### 14.2 主要约束

- 活动 ID、阶段 ID、参与者 ID和签到 ID 均为 UUID。
- 同一活动中的阶段位置唯一。
- 同一活动中的学号唯一。
- 同一活动中的手机号唯一。
- 同一参与者在同一阶段只有一条签到记录。
- 删除活动时，阶段和参与者级联删除，API 会先清理相关签到。
- 删除阶段时只写入 `deleted_at`，属于软删除。

### 14.3 数据持久化

Docker Compose 使用命名卷：

```text
radar-data
```

重新创建 Web 或 API 容器不会清空数据库。删除 Docker 命名卷、执行演示重置或手工清库才会造成数据丢失。

## 十五、输入规则和异常处理

### 15.1 活动输入

- 活动名称：2～80 个字符。
- 活动说明：最多 500 个字符。
- 状态：草稿、进行中或已结束。

### 15.2 阶段输入

- 阶段名称：2～50 个字符。
- 阶段说明：最多 300 个字符。
- 位置：正整数。

### 15.3 参与者输入

- 姓名：2～40 个字符。
- 学号：3～40 个字符。
- 手机号：符合数字格式要求。
- 学校：最多 80 个字符。
- 作品链接：最多 500 个字符且必须为完整 URL。

### 15.4 常见状态码

| 状态码 | 含义 | 常见原因 |
| --- | --- | --- |
| 200/201 | 成功 | 请求已处理 |
| 204 | 成功且无响应正文 | 退出、排序或删除成功 |
| 400 | 输入错误 | 缺少字段、URL 格式不正确、排序数据不完整 |
| 401 | 未认证 | 未登录或会话已过期 |
| 404 | 不存在 | 阶段不存在或后续签到未找到报名信息 |
| 409 | 状态冲突 | 活动未开放、唯一身份信息冲突 |
| 429 | 请求过多 | 一分钟内公开接口请求超过限制 |
| 500 | 服务器错误 | 数据库或服务异常 |

## 十六、常见问题与故障排查

### 16.1 手机扫码后提示“活动当前未开放”

原因：活动状态为“草稿”或“已结束”。

处理：管理员进入“活动设置”，将当前活动切换为“进行中”。

### 16.2 提示“未找到报名信息，请先完成报名”

可能原因：

- 扫描的是普通签到或作品提交二维码，不是报名二维码。
- 参与者尚未完成本活动报名。
- 填写的学号和手机号都与报名时不同。
- 报名和签到属于两个不同活动。

处理：先扫描当前活动的“活动报名”二维码，再使用相同学号或手机号签到。

### 16.3 提示“提交信息不完整或格式不正确”

检查：

- 姓名是否至少 2 个字符。
- 学号是否至少 3 个字符。
- 手机号是否只包含数字且长度正确。
- 学校是否过长。
- 作品链接是否以 `http://` 或 `https://` 开头。

### 16.4 报名后后台人数没有立即变化

后台每 30 秒自动刷新。可以点击右上角“刷新数据”立即更新。

同时确认：

- 后台选择的是正确活动。
- 手机扫描的是该活动对应二维码。
- 日期筛选没有排除刚刚产生的签到。

### 16.5 重复报名为什么人数不增加

这是设计行为。同一参与者在同一阶段只计数一次，避免重复扫描导致数据失真。

### 16.6 二维码指向 localhost 或内网地址

二维码使用当前后台页面的来源地址生成。

处理：

1. 不要通过 `localhost` 打开生产后台。
2. 使用公网 IP 或正式域名访问 `/admin`。
3. 再进入“签到入口”展示二维码。
4. 检查服务器 `.env` 中的 `PUBLIC_URL`。

### 16.7 手机打不开公网地址

检查：

- 华为云安全组是否放行 TCP 80。
- 服务器 Docker 容器是否运行。
- Web 容器是否映射 `0.0.0.0:80->80`。
- 手机是否能访问其他网络页面。
- 公网 IP 是否正确。

服务器执行：

```bash
docker compose ps
curl http://localhost/health
curl http://localhost/api/health
```

### 16.8 管理后台自动登录

浏览器中仍存在有效期最长 8 小时的管理员 Cookie。点击“退出登录”，或清除该网站 Cookie 后即可回到登录页。

### 16.9 管理员登录失败

检查：

- 用户名和密码是否与服务器 `.env` 一致。
- 修改 `.env` 后是否重新创建 API 容器。
- `JWT_SECRET` 是否满足长度要求。
- 是否设置了 `ADMIN_PASSWORD_HASH`；设置后明文 `ADMIN_PASSWORD` 不再用于校验。

重启：

```bash
docker compose up -d --force-recreate api
```

### 16.10 CSV 中文乱码

文件已包含 UTF-8 BOM。若 Excel 仍乱码，可使用“数据 → 从文本/CSV”，编码选择 UTF-8。

### 16.11 作品链接无法提交

必须填写完整 URL，例如：

```text
https://atomgit.com/用户名/仓库名
```

不要只填写 `atomgit.com/...` 或仓库名称。

### 16.12 某学号报名时提示数据冲突

同一活动中学号和手机号都必须唯一。如果新学号使用了已被其他学号绑定的手机号，数据库会拒绝写入。

管理员应核对参与者信息，避免不同人员共用手机号。当前后台没有直接修改或删除单个参与者的按钮，如确需修正，应由维护人员备份数据库后谨慎处理。

### 16.13 点击复制链接没有反应

HTTP 页面或内置浏览器可能限制剪贴板 API。可以：

- 使用 Edge/Chrome 打开后台并允许剪贴板权限。
- 直接用手机扫描二维码。
- 打开二维码对应页面后手工复制地址栏链接。

### 16.14 容器启动失败

```bash
docker compose config
docker compose logs --tail=200 api
docker compose logs --tail=200 db
docker compose exec db pg_isready -U radar -d radar
```

重点检查 `.env` 是否缺少 `POSTGRES_PASSWORD`、`JWT_SECRET`、`ADMIN_PASSWORD` 或 `PUBLIC_URL`。

## 十七、安全与隐私建议

### 17.1 密钥管理

- 不要将 `.env` 提交到仓库。
- 数据库密码、管理员密码和 JWT 密钥应各不相同。
- `JWT_SECRET` 建议使用 `openssl rand -hex 32` 生成。
- 不要在聊天、截图、答辩 PPT 或公开仓库中展示生产 SSH 密码。
- 仓库访问令牌应设置最小权限并定期轮换。

### 17.2 网络安全

- SSH 22 端口尽量只允许管理员固定 IP。
- 不要对公网放行 PostgreSQL 5432。
- 正式使用时配置 HTTPS。
- 定期更新服务器、Docker 镜像和 npm 依赖。

### 17.3 账号安全

- 当前系统是单管理员模式。
- 生产环境应更换演示密码。
- 公共电脑使用后必须退出登录。
- 管理员会话最长 8 小时。

### 17.4 参与者隐私

系统保存姓名、学号、手机号、学校和作品链接，属于应妥善保护的数据。

建议：

- 报名前告知数据用途和保存期限。
- 仅授权运营人员访问后台。
- 不公开导出的 CSV。
- 活动结束后按学校或赛事要求归档、脱敏或删除。
- 展示截图时遮挡真实手机号和学号。

## 十九、项目结构和技术架构

### 19.1 目录结构

```text
developer-retention-radar/
├─ apps/
│  ├─ api/                 # Express API、认证、迁移、演示数据和测试
│  └─ web/                 # React 学生端、管理后台和前端测试
├─ deploy/
│  └─ nginx.conf           # 生产反向代理和 SPA 路由配置
├─ docs/                   # 部署与答辩辅助文档
├─ skills/                 # 可复用训练营留存运营 Skill
├─ docker-compose.yml      # PostgreSQL、API、Web 编排
├─ Dockerfile              # API 和 Web 多阶段构建
├─ .env.example            # 环境变量模板
├─ README.md               # 项目概览
└─ 完整使用手册.md         # 本文档
```

### 19.2 请求链路

```text
手机/电脑浏览器
      │
      ▼
Nginx Web 容器（80/443）
      ├─ /            → React 静态页面
      └─ /api/*       → Express API 容器（3000）
                              │
                              ▼
                     PostgreSQL 容器（5432，仅内部网络）
```

### 19.3 关键设计

- 前后端同源部署，减少跨域和 Cookie 配置复杂度。
- Nginx 使用 SPA fallback，确保 `/admin` 和 `/checkin/:id` 刷新后仍能打开。
- API 启动时自动执行幂等迁移。
- 管理接口使用 JWT Cookie 保护。
- 公开接口进行输入校验和每分钟限流。
- 阶段和参与者使用 UUID，避免可预测递增 ID。
- 数据库唯一约束保障身份和签到去重。
- 数据库命名卷保障容器重建后数据持久化。

## 附录：最重要的操作提醒

1. 新建活动后一定要切换为“进行中”，否则无法报名。
2. 参与者必须先扫描“活动报名”，才能完成后续阶段。
3. 后续签到应继续使用报名时的学号或手机号。
4. 重复签到成功但不重复计数，这是正常行为。
5. 最终提交必须填写带 `http://` 或 `https://` 的完整链接。
6. 后台数据每 30 秒自动刷新，也可手动点击“刷新数据”。
7. “重置演示数据”会清空整个系统的全部活动和数据。
8. 二维码必须从公网地址打开的后台生成和展示。
9. 活动结束后先导出数据、备份数据库，再切换为“已结束”。
10. 正式使用前必须更换演示管理员密码并配置 HTTPS。
